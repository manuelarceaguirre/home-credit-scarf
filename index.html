1. Overview

1.1 Project Goal

I implemented a borrower segmentation pipeline on the Home Credit Default Risk dataset. My main objective was not just prediction. It was representation quality for segmentation: can I build borrower clusters that are geometrically coherent and risk-differentiated enough to support strategy and presentation-level interpretation?

The core comparison in this project is:

  • Classical baseline: PCA + clustering
  • Novel method: SCARF contrastive representation learning + clustering

I intentionally used the same split-first protocol and as many shared downstream evaluation steps as possible so that the comparison is fair and interpretable.

1.2 Why This Matters for Us

For this class project, we are expected to communicate both methodological rigor and practical interpretation. A pure classifier can give good AUC without giving us segments that are easy to discuss in a portfolio narrative. In contrast, segmentation quality depends on latent geometry. I designed the full notebook around that distinction.

In plain terms, I asked: can a self-supervised encoder learn a borrower space where similar borrowers are close, corrupted views stay consistent, and clustering has tighter structure than linear PCA space?

1.3 Snapshot of Final Quantitative Story

At a high level, PCA won on linear probe AUC (0.744918 vs 0.642574), while SCARF strongly won on silhouette and IV-oriented cluster quality metrics. That tradeoff is the central narrative we should present: linear separability versus segmentation geometry.

2. The Papers

2.1 SCARF (Bahri et al., ICLR 2022)

Primary paper: SCARF: Self-Supervised Contrastive Learning using Random Feature Corruption.

The key idea is elegant for tabular data. Images have natural augmentations like crop, blur, or color jitter. Tabular data does not have an obvious equivalent that preserves semantics. SCARF solves this by creating positive pairs through random feature corruption: for each sample x, it randomly selects a subset of features and replaces them with feature values from donor rows. It then trains with contrastive loss so the representation of x and its corrupted view stay close, while being far from other rows in the batch.

In formula form, NT-Xent/InfoNCE for a positive pair (i, j) looks like:

Loss(i,j) = -log( exp(sim(z_i,z_j)/tau) / sum_{k != i} exp(sim(z_i,z_k)/tau) )

where sim is cosine similarity and tau is temperature.

Why this is smart for credit tabular data: missingness, sparse categoricals, and heterogeneous scales all make direct Euclidean relationships brittle in raw feature space. SCARF learns invariances to realistic feature perturbations without labels.

2.2 ReConTab (arXiv 2023)

Reference: ReConTab.

I used ReConTab mainly as contextual evidence that contrastive objectives for tabular representations are an active and credible direction. It reinforces the idea that robustness to corruption and structure-aware representation learning can improve downstream organization of tabular cohorts, including unsupervised analyses.

2.3 TCCL (arXiv 2025)

Reference: Tabular Continual Contrastive Learning (TCCL).

I cite this as a recent direction showing the tabular contrastive ecosystem is still advancing, especially around OOD robustness and transfer in changing distributions. I did not implement TCCL here, but I used it to motivate why SCARF is not an isolated idea and why this project is aligned with current research trajectories.

2.4 Why I Chose SCARF Over Alternatives

I considered several alternative families before committing:

  • VAEs: I often see blurry latent organization for structured tabular risk data unless heavily tuned. There is also posterior collapse risk and unclear cluster semantics without strong regularization choices.
  • Deep clustering (DEC/IDEC): Can work, but is sensitive to initialization, pretraining strategy, and cluster-assignment sharpening behavior. Mode collapse risk is real if training dynamics are not carefully managed.
  • Graph-based approaches: On 300K+ rows, graph construction and nearest-neighbor graph updates become expensive quickly, and memory/computation constraints become part of the method rather than the problem.

SCARF gave me a clean middle ground: tabular-specific augmentation logic, efficient minibatch training, and direct compatibility with standard downstream clustering.

I measured missingness on train only and added binary indicator columns _MISSING for features with missing ratio greater than 5%. In this run, that produced 57 missingness indicators.

  2. Apply dummy encoding.
  3. Reindex test to train columns.

8. SCARF Implementation (Most Detailed)

8.1 ScarfDataset and Corruption Logic

The dataset returns two views per item: original row and corrupted row. Corruption rate is 60% over logical features (numeric singletons + categorical groups). Each selected logical feature gets values replaced by values from a random donor row. Grouped one-hot features are swapped atomically.

Full __getitem__ method:

def __getitem__(self, idx):
    x = self.data[idx].clone()
    x_corrupt = x.clone()

    # Select logical features to corrupt.
    perm = torch.randperm(self.n_logical)[: self.n_corrupt]

    # Gather all column indices to corrupt and their donors in one pass.
    cols_to_corrupt = []
    donors_for_cols = []

    for p in perm.tolist():
        donor = torch.randint(0, self.n_samples, (1,)).item()
        if p < len(self.numeric_indices):
            cols_to_corrupt.append(self.numeric_indices[p])
            donors_for_cols.append(donor)
        else:
            group_idx = p - len(self.numeric_indices)
            for c in self.cat_groups[group_idx]:
                cols_to_corrupt.append(c)
                donors_for_cols.append(donor)

    # Vectorized assignment
    if cols_to_corrupt:
        col_idx = torch.tensor(cols_to_corrupt, dtype=torch.long)
        donor_idx = torch.tensor(donors_for_cols, dtype=torch.long)
        x_corrupt[col_idx] = self.data[donor_idx, col_idx]

    return x, x_corrupt

I vectorized the final assignment so I do not loop tensor writes column by column. This keeps data loading reasonably fast even at batch size 512.

8.4 NT-Xent Loss (Full Class)

class NTXentLoss(nn.Module):
    # Standard NT-Xent / InfoNCE over positive pairs in a batch.

    def __init__(self, temperature=0.5):
        super().__init__()
        self.temperature = temperature

    def forward(self, z_i, z_j):
        batch_size = z_i.shape[0]

        z_i = F.normalize(z_i, dim=1)
        z_j = F.normalize(z_j, dim=1)

        z = torch.cat([z_i, z_j], dim=0)  # [2N, D]
        sim = torch.matmul(z, z.T) / self.temperature  # [2N, 2N]

        # mask self-similarity
        eye_mask = torch.eye(2 * batch_size, device=sim.device, dtype=torch.bool)
        sim = sim.masked_fill(eye_mask, -1e9)

        # positive similarities are offset by batch_size
        pos = torch.cat([
            torch.diag(sim, batch_size),
            torch.diag(sim, -batch_size),
        ], dim=0)

        # denominator includes all non-self entries
        denom = torch.logsumexp(sim, dim=1)

        loss = -pos + denom
        return loss.mean()

Step by step:

  1. L2 normalize projected vectors.
  2. Concatenate original and corrupted projections to shape [2N, D].
  3. Compute all-pairs cosine similarity matrix scaled by temperature (0.5).
  4. Mask self-similarity terms on the diagonal.
  5. Positive pairs are exactly diagonal offsets by batch size.
  6. Compute -positive + logsumexp(all_non_self) and average.

8.5 Training Loop and Configuration

Configuration used:

  • Epochs: 30
  • Batch size: 512
  • Optimizer: Adam
  • Learning rate: 1e-3
  • Weight decay: 1e-5
  • Scheduler: CosineAnnealingLR
  • Corruption rate: 0.6
  • Temperature: 0.5

Key function:

def train_scarf(encoder, projector, dataloader, epochs=50, lr=1e-3, weight_decay=1e-5, temperature=0.5):
    encoder.train()
    projector.train()

    criterion = NTXentLoss(temperature=temperature)
    optimizer = torch.optim.Adam(
        list(encoder.parameters()) + list(projector.parameters()),
        lr=lr,
        weight_decay=weight_decay,
    )
    scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=epochs)

    history = []
    for epoch in range(1, epochs + 1):
        epoch_losses = []
        for x, x_corrupt in dataloader:
            x = x.to(device, non_blocking=True)
            x_corrupt = x_corrupt.to(device, non_blocking=True)
            optimizer.zero_grad(set_to_none=True)

            h_i = encoder(x)
            h_j = encoder(x_corrupt)
            z_i = projector(h_i)
            z_j = projector(h_j)

            loss = criterion(z_i, z_j)
            loss.backward()
            optimizer.step()
            epoch_losses.append(loss.item())

        scheduler.step()
        history.append(float(np.mean(epoch_losses)))

    return history

8.6 Embedding Extraction and Why Raw Encoder Outputs

I extracted raw encoder outputs (not normalized):

@torch.no_grad()
def extract_embeddings(encoder, X, batch_size=2048):
    encoder.eval()
    dataset = TensorDataset(torch.tensor(X, dtype=torch.float32))
    loader = DataLoader(dataset, batch_size=batch_size, shuffle=False, num_workers=0)

    all_emb = []
    for (batch,) in loader:
        batch = batch.to(device, non_blocking=True)
        emb = encoder(batch)
        # We extract unnormalized encoder embeddings. NT-Xent normalizes the
        # projection head output, NOT the encoder. Euclidean K-Means on raw
        # encoder outputs is intentional: magnitude encodes feature intensity.
        all_emb.append(emb.cpu().numpy())

    return np.vstack(all_emb)

This was intentional. Normalizing encoder outputs would remove norm information that can encode profile intensity useful for Euclidean K-Means.

11. What the Results Mean (Honest Discussion)

11.1 Why PCA Won AUC

PCA optimizes variance in a linear subspace. A linear probe is also a linear decision boundary. That geometric match explains why PCA can produce better linear probe AUC even if it does not produce the cleanest cluster geometry. In our run, PCA AUC was 0.744918 versus 0.642574 for SCARF.

I do not interpret that as SCARF being weak overall. I interpret it as objective mismatch: SCARF was not trained to maximize linear separability of default labels.

11.2 Why SCARF Won Silhouette and IV

SCARF explicitly optimizes local instance discrimination under corruption invariance. That tends to organize neighborhoods more tightly in representation space. For unsupervised segmentation, that often translates into better silhouette and stronger risk differentiation once points are clustered.

In this run, silhouette roughly doubled from 0.179730 to 0.385522. IV at chosen k also increased substantially from 0.025499 to 0.062542.

11.3 Shared-k IV Is the Important Fairness Check

Because methods selected different best k values (2 vs 9), I added shared-k evaluation at k=2,4,6,8. This is crucial for an honest comparison. At k=2, SCARF was slightly below PCA (0.024636 vs 0.025499), but from k=4 onward SCARF was consistently higher. That suggests SCARF benefits more from finer granularity in segmentation.

11.4 Practical Presentation Narrative

For our team presentation, I suggest we say this clearly: if the task is a linear supervised default probe, PCA is stronger in this run. If the task is unsupervised segmentation quality and risk stratification through clusters, SCARF gives more structure and higher IV, especially beyond coarse binary segmentation.

This framing is not defensive; it is technically honest and aligned with each method's objective.

11.5 What Would Likely Improve SCARF Next

  • Train longer than 30 epochs.
  • Tune corruption rate around 0.4 to 0.8.
  • Tune temperature around 0.1 to 1.0.
  • Increase encoder width/depth.
  • Add light supervised fine-tuning if downstream objective includes classification.
  • Expand seed sweeps for robustness and confidence intervals.

12. Assumptions I Made

I want these explicit so we can discuss limitations without surprises.

  1. IID rows: I assumed borrowers are independent samples for random split validity.
  2. Median imputation and MAR-like behavior: I assumed missing numeric values are recoverable enough with median plus indicators.
  3. 365243 sentinel meaning: I assumed this value truly means unknown employment duration, not an extreme but valid observation.
  4. Corruption rate 0.6 is reasonable: I used 60% without tuning search.
  5. Temperature 0.5 is reasonable: I used a standard NT-Xent default without calibration.
  6. 30 epochs are enough: I assumed near-converged representation quality in this budget.
  7. 64-dimensional embedding is enough: I assumed this dimension balances capacity and noise.
  8. Euclidean distance for K-Means is appropriate: I assumed cluster geometry in encoder space is meaningful under Euclidean metric.
  9. 90% PCA variance threshold is a fair baseline setting: I assumed it is a conventional and adequate reduction target.
  10. Ward linkage suitability: I assumed Ward hierarchical behavior is informative even if cluster shapes are not perfectly spherical.
  11. Single-dataset evaluation suffices for project scope: I did not run external dataset validation.
  12. No SCARF hyperparameter tuning beyond defaults: I assumed base config is enough for method demonstration.
  13. Bootstrap n=10 is acceptable for uncertainty sketch: I assumed this is enough for a class project confidence estimate, not publication-grade intervals.
  14. 5% threshold for missing indicators: I assumed this cutoff captures important missingness signals without too many noisy indicators.
